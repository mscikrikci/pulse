## 1. Product Vision

Pulse bridges the gap between raw Apple Health data and meaningful behavioral change. It operates across three time horizons simultaneously:

- **Today** — Am I recovered? What should I prioritize right now?
- **This week** — Are my habits consistent enough to expect progress?
- **The journey** — Am I trending toward my goals? What is the highest-leverage behavior to focus on?

### Core Design Principles

**Outcome-oriented, not activity-oriented.** Goals are set on physiological outcomes (HRV, resting heart rate, sleep respiratory rate). Activity metrics (steps, calories) are inputs that serve those outcomes — they drive nudges, not goals.

**Personal baseline over population norms.** All evaluations compare the user against their own rolling baseline. A "good" HRV is relative to that individual, not a population average.

**Minimal manual input.** The app infers as much as possible from HealthKit. Manual input is limited to one daily check-in card (alcohol + subjective events) and goal-setting screens.

**Conversational, not dashboard.** Insights are delivered through natural language generated by an LLM. Data visualizations support the narrative rather than replace it.

**Not medical advice.** All suggestions are framed as performance optimization protocols sourced from Huberman Lab research. No diagnoses, no clinical claims.

**Context-aware suggestions.** The system tracks what the user has already done today (steps, activity, reported events) and ensures it never recommends completing something already completed.

## 4. Data Layer

### 4.1 HealthKit Queries

All queries are foreground-only, triggered on app open or user action. No background delivery.

| Metric | HealthKit Type | Query Window | Aggregation |
|---|---|---|---|
| Sleep duration | `sleepAnalysis` | Last night (6pm–12pm) | Sum of asleep stages (core + deep + REM) |
| Sleep efficiency | `sleepAnalysis` | Last night | Asleep duration / time in bed |
| Wake time | `sleepAnalysis` | Last night | Latest end date of any sleep sample |
| HRV (SDNN) | `heartRateVariabilitySDNN` | Last night (9pm–9am) | Average of samples |
| Resting Heart Rate | `restingHeartRate` | Today | Latest daily sample |
| Respiratory Rate | `respiratoryRate` | Last night (9pm–9am) | Average of samples |
| Active Calories (yesterday) | `activeEnergyBurned` | Yesterday midnight–midnight | Sum |
| Steps (yesterday) | `stepCount` | Yesterday midnight–midnight | Sum |
| Active Calories (today) | `activeEnergyBurned` | Today midnight–now | Sum (real-time) |
| Steps (today) | `stepCount` | Today midnight–now | Sum (real-time) |

**Real-time today data** (`todayCalories`, `todaySteps`, `wakeTime`) is fetched on every `fetchTodaySummary()` call and used by the context builder for time-aware suggestions. This ensures the LLM knows about activity done _today_, not just yesterday's stored totals.

**Null handling rule:** If a metric is unavailable, it is emitted to the LLM as:
```
HRV: not available (Watch not worn or data not recorded)
```
Never omit the field, never pass zero or empty string.

### 4.2 HealthSummary struct

The primary data object passed between HealthKit and all downstream services:

```swift
struct HealthSummary: Codable {
    let date: Date
    var hrv: Double?                    // ms, last night average
    var restingHR: Double?              // bpm
    var sleepHours: Double?             // total hours asleep
    var sleepEfficiency: Double?        // 0.0–1.0
    var respiratoryRate: Double?        // breaths/min during sleep
    var activeCalories: Double?         // yesterday's total kcal
    var steps: Int?                     // yesterday's total
    var wakeTime: Date?                 // derived from latest sleep sample end
    var todayCalories: Double?          // today midnight–now kcal
    var todaySteps: Int?                // today midnight–now steps
}
```

### 4.3 Trend Store

**File:** `Documents/trend_store.json`
**Updated:** Every app open via `store.update(with: summary)`
**Actor-based** for thread-safe writes.

```json
{
  "last_updated": "2026-03-03T08:14:00Z",
  "baseline_data_days": 24,
  "baselines": {
    "hrv_7day_avg": 61.2,
    "hrv_30day_avg": 58.3,
    "resting_hr_30day_avg": 52.1,
    "sleep_hours_7day_avg": 7.0,
    "sleep_efficiency_30day_avg": 0.84,
    "respiratory_rate_30day_avg": 14.5,
    "active_calories_7day_avg": 580,
    "steps_7day_avg": 7800
  },
  "history": [
    {
      "date": "2026-03-02",
      "hrv": 61,
      "resting_hr": 52,
      "sleep_hours": 7.4,
      "sleep_efficiency": 0.87,
      "respiratory_rate": 14.2,
      "active_calories": 620,
      "steps": 8200,
      "alcohol_reported": false,
      "events": ["high_stress"]
    }
  ]
}
```

**`baseline_status` values:**
- `cold` — fewer than 7 days of data. LLM instructed to use population reference ranges and qualify all statements.
- `building` — 7–29 days. Personal baselines exist but may still shift.
- `established` — 30+ days. Full personal baseline, normal operation.

**DayEntry fields:**
- `alcohol_reported: Bool?` — set by daily log card, recorded when user taps Yes/No
- `events: [String]?` — user-reported subjective events for that day (e.g. `["high_stress", "fatigue"]`)

### 4.4 Goal Store

**File:** `Documents/goal_store.json`
**Updated:** On goal save or activity alert configuration.

Stores `GoalDefinition` and `ActivityAlert` arrays. See §9 for full goal model.

### 4.5 Progress Store

**File:** `Documents/progress_store.json`
**Updated:** Weekly (Sunday) and on phase advancement.

Stores per-goal phase history, weekly snapshots, and overall pace assessment.

### 4.6 Morning Card Store

**File:** `Documents/morning_card.json`
**Updated:** Whenever a morning card is successfully generated.
**Pattern:** Date-keyed (`"yyyy-MM-dd"`). `loadIfToday()` returns the card only if it was generated on today's date.

This enables persistent morning cards — the card survives app restarts and is shown immediately on reopen without re-generating. Once per day generation only.

### 4.7 Daily Log State (UserDefaults)

Simple date-keyed flags stored in `UserDefaults`:

| Key | Value | Purpose |
|---|---|---|
| `"dailyLogDate"` | `"yyyy-MM-dd"` | Whether the daily log form was submitted today |
| `"dailyLogEvents"` | `[String]` | Events selected in today's log |
| `"alcoholCheckedDate"` | `"yyyy-MM-dd"` | Whether alcohol was answered today |

Both are checked on `bootstrap()` and restored to the `TodayViewModel` state so the UI reflects the correct submitted state without re-prompting.

## 5. Feature: Today Tab

The primary daily surface. Shown on the `Today` tab (first tab in the nav bar).

### 5.1 Card Stack (top to bottom)

1. **Safe zone alert banner** — shown if any metric is in warning or alert state (§14)
2. **Morning card** — readiness report generated by LLM (§5.2)
3. **Mid-day check-in card** — appears after morning card is available (§6)
4. **Daily log card** — appears once health data is loaded; collapses after submission (§7)
5. **Weekly review card** — always visible; prompts generation or shows existing review

### 5.2 Morning Card

The morning card is generated once per day on first app open (or on explicit "Generate" tap). It is persisted to `MorningCardStore` and shown immediately on subsequent opens within the same calendar day.

**States:**
- **Loading** — skeleton placeholder shown while LLM call is in flight
- **Generated** — full card rendered by `MorningCardView`
- **Fallback** — if JSON parse fails, raw LLM text shown with "Retry" button
- **Error** — network/API failure; error message + "Retry" button
- **Not generated** — "Generate Morning Card" button shown (first open of the day)

**Morning Card fields:**

| Field | Description |
|---|---|
| `readiness_level` | `"high"` / `"medium"` / `"low"` / `"alert"` |
| `headline` | Punchy 8-word summary |
| `summary` | 2–3 sentences grounded in specific numbers |
| `work_suggestion` | One sentence on cognitive/physical work fit |
| `protocols` | 2–3 selected from corpus with per-protocol reasoning |
| `avoid_today` | List of things to avoid today |
| `one_focus` | The single most impactful action today |
| `goal_note` | How today relates to active goal journey |

**Readiness badge colors:**
- `high` → green
- `medium` → yellow
- `low` → orange
- `alert` → red

**Protocol rows** are expandable — tap to show the reasoning for why this protocol was selected today. Protocol IDs come verbatim from `protocols.json`; the LLM is explicitly instructed not to invent IDs.

### 5.3 What the Morning Card Knows

The context sent to the LLM for morning card generation includes:

- Last night's HRV, sleep, resting HR, respiratory rate (with baselines and deltas)
- Yesterday's active calories and steps (with 7-day averages)
- Current session: time of day, how long awake, today's real-time steps and calories so far
- Today's reported subjective events (if daily log was submitted before generation)
- Active goals with mode, target, phase, and current pace
- 2–5 pre-matched protocols from the corpus relevant to today's condition flags
- Baseline status (cold / building / established)

## 6. Feature: Mid-Day Check-In

Shown below the morning card once it exists. Allows the user to compare their current state to the morning snapshot.

### 6.1 States

**Trigger button state:**
- "How's the day going?" prompt with "Check In" button
- Tapping the button runs `generateCheckIn()` which:
  1. Fetches a **fresh** `HealthSummary` (not the cached morning snapshot) — ensures steps/calories from after morning app open are counted
  2. Builds check-in context comparing morning card to current state
  3. Calls LLM with `checkInInstruction` (haiku model, 512 tokens)
  4. Parses response into `CheckInResponse`

**Result state:**
- Headline
- Observation (what changed since morning — specific to numbers)
- Suggestion (concrete immediate action — **not** something already done today)
- "Explore in Chat →" button
- "Refresh" button

### 6.2 Activity Completion Awareness

The check-in context explicitly signals completed activities to prevent the LLM from suggesting something the user already did:

```
Estimated walking: ~32 min of movement (in 1.1h since waking)
IMPORTANT: The user has already completed ~32 min of walking/movement since waking.
Do NOT suggest another walk, outdoor stroll, or morning movement — that protocol is DONE.
```

The step-to-duration estimate uses `steps ÷ 100 ≈ minutes` (100 steps/min average walking pace). This fires when estimated duration ≥ 10 minutes.

The check-in context also lists the morning card's suggested protocols by ID:
```
Morning protocols suggested: morning_sunlight_movement, nsdr_yoga_nidra
```

So the LLM can cross-reference: "sunlight walk was suggested AND ~30 min of walking has been done → suggest NSDR instead."

### 6.3 "Explore in Chat →" Deep Link

Tapping this button passes a `PendingChatContext` from Today tab to the Chat tab:

```swift
struct PendingChatContext: Equatable {
    let uiPreamble: String        // shown as cosmetic assistant bubble in Chat UI
    let checkInContext: String    // injected into LLM context (not shown in UI)
    let userMessage: String       // auto-sent as first user message
}
```

- `uiPreamble` — formatted markdown of the check-in observation and suggestion, shown as an assistant bubble in Chat
- `checkInContext` — observation + suggestion text injected into the LLM context under `=== MID-DAY CHECK-IN CONTEXT ===` so the model has full awareness
- `userMessage` — the `chat_prompt` field from `CheckInResponse`, auto-sent as the first user message

The preamble bubble is marked `isUIOnly: true` on the `ChatViewModel.Message` struct — it is shown in the UI but **never sent to the Anthropic API** (the API requires the first message to be from the user role).

## 7. Feature: Daily Subjective Log

A daily check-in card that collects qualitative context the LLM cannot observe from HealthKit alone. Appears in the Today tab whenever health data is loaded. Submitted once per day.

### 7.1 Card Fields

**Alcohol (yesterday):**
- "Any alcohol yesterday?" — Yes / No button pair
- Tapping one highlights that button; the other fades
- Required before the Save button enables

**Subjective events (today):**
Multi-select chip grid. Any number can be selected.

| Event | Raw Value | Maps to Protocol Flag |
|---|---|---|
| High Stress | `high_stress` | `high_stress` |
| Fatigue | `fatigue` | `poor_sleep` (fatigue → recovery protocols) |
| Sadness | `sadness` | `high_stress` (same stress-reduction tools apply) |
| Argument | `argument` | `high_stress` |
| Anger | `anger` | `high_stress` |
| Health Issue | `health_issue` | `possible_illness` |

### 7.2 Persistence & Submission

On "Save Log":
1. Alcohol result written to `TrendStore.recordAlcohol()`
2. Events array written to `TrendStore.recordEvents()`
3. Today's date key written to `UserDefaults` (`dailyLogDate`, `dailyLogEvents`, `alcoholCheckedDate`)
4. Card collapses to compact "Daily log saved" summary showing events logged

On next app open: `bootstrap()` checks `dailyLogDate` against today. If matching, `dailyLogSubmitted = true` and the compact summary is shown immediately — no re-prompting.

### 7.3 How Events Feed Into the System

Events flow into three places:

1. **ProtocolMatcher** — user events are mapped to existing trigger flags (`high_stress`, `poor_sleep`, `possible_illness`) and added to the flag set. This causes relevant protocols (physiological sigh, NSDR, journaling, illness rest, etc.) to appear in the morning card's protocol suggestions.

2. **Context builder** — added to the "Today's Log" block:
   ```
   Today's Log:
   - Reported: high_stress, fatigue
   ```
   This appears in every morning card context, chat context, and check-in context.

3. **Check-in context** — also included in `buildCheckInContext()`:
   ```
   Reported today: High Stress, Fatigue
   ```

## 8. Feature: Progress Tab

Scrollable list of active goals, each shown as a `GoalProgressCardView`. Named `GoalProgressTab` in code to avoid naming collision with SwiftUI's built-in `ProgressView`.

### 8.1 Target Mode Goal Card

Shows:
- Metric name, current 7-day average, gap to target
- Phase timeline bar: Phase 1 → Phase 2 → Phase 3, with current phase highlighted and phase focus text + sub-target range
- Pace badge: **Ahead** / **On Track** / **Behind** / **Stalled** (color-coded)
- Projected completion date (extrapolated from current pace)
- Weekly sparkline: Canvas-based line chart of all weekly snapshots (no Charts framework)

### 8.2 Maintain Mode Goal Card

Shows:
- Metric name, current value, baseline
- Safe zone monitoring status (Normal / Warning / Alert)
- Latest value and whether it is within acceptable range

### 8.3 Cold Baseline State

When `baselineDataDays < 7`, the pace assessment is not shown. Instead: "Building baseline — pace assessment available after 7 days of data."

### 8.4 Sparkline

Custom `Canvas`-based SwiftUI view. Draws:
- Line connecting weekly averages
- Dots at each data point
- Optional fill gradient below the line
- Target value as a dashed horizontal rule

No external dependencies.

## 9. Feature: Goals Tab

Goal management screen. Active goals shown as a list with inline safe zone status. Setup is per-metric.

### 9.1 Goal Modes

**Target mode** — user wants to improve a metric to a specific value within an optional timeframe. App generates a phased progression plan, tracks weekly pace, and evaluates phase advancement automatically.

**Maintain mode** — user wants to keep their current baseline. App monitors for safe zone violations. No phase structure.

### 9.2 Goal Metrics

Three physiological outcome metrics:

| Metric | Unit | Direction | Safe Zone Type |
|---|---|---|---|
| Heart Rate Variability | ms | Higher is better (↑) | Relative (% of 30-day avg) |
| Resting Heart Rate | bpm | Lower is better (↓) | Absolute (fixed bpm thresholds) |
| Sleep Respiratory Rate | breaths/min | Lower is better (↓) | Absolute (fixed br/min thresholds) |

Direction is shown explicitly in the Goals UI so users know which way to aim when entering a target value.

### 9.3 HRV Phase Model

Pre-authored for target-mode HRV goals. Phase template selected based on gap size (target − baseline). Three phases:

| Phase | Duration | Sub-target | Focus |
|---|---|---|---|
| 1 — Stabilize | Weeks 1–3 | baseline + 1–2ms | Consistent sleep timing, alcohol elimination, morning sunlight |
| 2 — Build | Weeks 4–8 | baseline + 4–7ms | Zone 2 cardio 2–3×/week, daily NSDR, cold exposure |
| 3 — Optimize | Weeks 9–14 | Full target | Training load management, meal timing, minimal alcohol |

Phase advances automatically when the 7-day average exceeds the sub-target upper bound for 2 consecutive weekly snapshots.

### 9.4 Activity Alerts

Configurable per-metric (steps and active calories). Each alert has:
- Daily target (informational)
- Alert-below threshold (triggers 5pm notification)
- Optional link to an outcome goal (provides contextual rationale in notification copy)
- Enabled/disabled toggle

### 9.5 Safe Zone Thresholds

**HRV (relative):**
- Warning: today < 30-day avg × 0.85
- Alert: today < 30-day avg × 0.75

**Resting HR (absolute):**
- Warning: ≥ 57 bpm
- Alert: ≥ 60 bpm

**Respiratory Rate (absolute):**
- Warning: ≥ 17 breaths/min
- Alert: ≥ 19 breaths/min

### 9.6 Goal Conflict Detection

On save, `GoalConflictDetector` checks known conflict patterns and shows a warning sheet (does not block saving):

- High calorie target (> 800 kcal/day) + active HRV improvement goal
- Aggressive RHR reduction (> 10 bpm in < 8 weeks)

## 10. Feature: Chat Tab

Ad-hoc conversational interface. Every message automatically has full health context injected — the user asks questions in plain language, the LLM has access to all the same data as the morning card.

### 10.1 Context Per Message

Before each `send()`:
1. `HealthSummary` is **refreshed from HealthKit** — ensures each message reflects current steps/calories, not a stale snapshot from when the tab was first loaded
2. `ContextBuilder.buildChatContext()` is called with the fresh summary, trend store data, goals, and today's logged events
3. Today's subjective events (from UserDefaults) are also loaded and included

### 10.2 Message Architecture

```swift
struct Message: Identifiable {
    let id: UUID
    let role: Role          // .user | .assistant
    let content: String
    var isUIOnly: Bool      // if true: shown in UI, never sent to Anthropic API
}
```

`isUIOnly` is set `true` on the cosmetic preamble bubble injected by "Explore in Chat →". This allows a check-in summary to appear as an assistant bubble without violating the Anthropic API constraint that the messages array must start with a user message.

**History sent to API:** `messages.dropLast().filter { !$0.isUIOnly }` — all non-UI-only messages before the current user message, in `[{role, content}]` format.

### 10.3 Model Selection

Chat questions are automatically routed to the appropriate model:

```swift
let isComplex = context.contains("week") || context.contains("month") ||
                context.contains("trend") || context.contains("goal progress")
let model = isComplex ? analysisModel : defaultModel
```

Haiku for simple daily questions; Sonnet for multi-week trend analysis.

### 10.4 Suggested Questions

Shown when the conversation is empty:
- "How is my HRV trending this week?"
- "Am I getting enough sleep?"
- "What should I prioritize for recovery today?"
- "How am I progressing toward my goals?"

### 10.5 Keyboard Management

- `@FocusState` on the text field
- `.scrollDismissesKeyboard(.immediately)` on the scroll view — scrolling the message list dismisses the keyboard so the user can read responses and navigate to other tabs
- Keyboard toolbar "Done" button as explicit dismiss

### 10.6 Pending Chat Deep Link

When navigating from Today tab → Chat via "Explore in Chat →":
- `PendingChatContext` is set in `ContentView` state and passed as a binding to `ChatView`
- `ChatView` has both `.onAppear` and `.onChange` handlers — `.onAppear` handles the case where Chat tab is being shown for the first time (SwiftUI batches the tab switch and context set in the same state update, so `onChange` doesn't fire for a first-load); `.onChange` handles subsequent navigations

## 11. Protocol Corpus (RAG)

### 11.1 File

`Bundle/protocols.json` — bundled with the app, ~50 protocols at launch. Read-only at runtime. Updated via app releases.

### 11.2 Protocol Structure

```json
{
  "id": "physiological_sigh",
  "title": "Physiological Sigh",
  "category": "stress_reduction",
  "tags": ["stress", "hrv", "nervous_system"],
  "trigger_conditions": ["high_stress", "low_hrv", "elevated_rhr"],
  "duration_minutes": 1,
  "effort": 0,
  "timing": "anytime",
  "protocol": "Double inhale through the nose...",
  "why_it_works": "Collapsed alveoli trigger a stress response...",
  "source": "Huberman Lab — Controlling Stress in Real Time",
  "phase_relevance": ["hrv_phase_1", "hrv_phase_2"]
}
```

`effort` is an integer 0–5 (0 = no effort, 5 = high effort). Used for sorting: low-effort protocols are prioritized on low-readiness days.

### 11.3 Protocol Categories (launch corpus)

| Category | Count | Examples |
|---|---|---|
| Stress & Nervous System | 10 | Physiological sigh, box breathing, NSDR, visual field dilation, journaling |
| Sleep & Circadian | 10 | Morning sunlight, consistent wake time, temperature for sleep onset, no light after 10pm, caffeine cutoff |
| HRV & Recovery | 8 | Zone 2 cardio, deload week, HRV-guided training load, cold/hot contrast, alcohol-free baseline |
| Morning Routine | 8 | Morning sunlight + movement combo, cold shower, delayed caffeine, morning hydration, no phone first 30 min |
| Focus & Cognitive | 7 | 90-min ultradian blocks, visual focus narrowing, NSDR between blocks, binaural beats, cold for focus |
| Movement | 7 | Zone 2 3×/week, HIIT 1–2×/week max, non-exercise activity (NEAT), VO2max intervals (4×4) |

### 11.4 Retrieval Logic

Rule-based (no vector embeddings). `ProtocolMatcher`:

1. Computes condition flags from `HealthSummary + Baselines + userEvents`
2. Filters protocols where any `trigger_conditions` entry is in the flag set
3. Sorts by effort (ascending for low/alert readiness, descending otherwise)
4. Returns top 5

**Condition flag mapping:**

| Flag | Trigger Criteria |
|---|---|
| `low_hrv` | HRV > 15% below 7-day average |
| `very_low_hrv` | HRV > 25% below 7-day average |
| `elevated_rhr` | RHR > 5 bpm above 30-day average |
| `poor_sleep` | Sleep < 6.5h OR efficiency < 75% OR sleep data absent |
| `elevated_respiratory_rate` | Resp rate > 17 breaths/min |
| `high_stress` | Derived from `low_hrv + elevated_rhr` OR user reported `high_stress / sadness / argument / anger` |
| `possible_illness` | Derived from `elevated_resp + elevated_rhr` OR user reported `health_issue` |
| `peak_readiness` | HRV > 10% above 7-day avg AND sleep ≥ 7h |
| `well_recovered` | No negative flags present (fires even when HRV data is absent) |

The `well_recovered` default ensures users without Apple Watch (no HRV data) still get matched protocols. Previously this flag required HRV data, causing zero protocol matches on watch-free days.

**User event → flag mapping:**

| Reported Event | Protocol Flag |
|---|---|
| `high_stress`, `sadness`, `argument`, `anger` | `high_stress` |
| `fatigue` | `poor_sleep` |
| `health_issue` | `possible_illness` |

## 12. LLM Integration

### 12.1 AnthropicClient

`actor AnthropicClient` — thread-safe, direct REST calls to `https://api.anthropic.com/v1/messages`.

Required headers:
```
x-api-key: {apiKey}
anthropic-version: 2023-06-01
content-type: application/json
```

Response parsing: `response.content[0].text`. Markdown fence stripping applied automatically (`stripMarkdownFences()`) since models sometimes wrap JSON in ` ```json ` blocks despite being instructed not to.

### 12.2 API Calls by Feature

| Feature | Method | Model | Max Tokens |
|---|---|---|---|
| Morning card | `completeMorningCard()` | haiku | 1024 |
| Mid-day check-in | `completeCheckIn()` | haiku | 512 |
| Weekly review | `completeWeeklyReview()` | sonnet | 2048 |
| Chat (simple) | `completeChat()` | haiku | 1024 |
| Chat (complex) | `completeChat()` | sonnet | 1024 |

### 12.3 System Prompt

Applied to every API call. Defines the assistant's identity, role, constraints, and tone. Never constructed dynamically.

Key constraints:
- Never suggest training through poor HRV or sleep debt
- Always prioritize recovery over fitness goals when in conflict
- Acknowledge data limitations explicitly when baseline < 14 days
- Distinguish "behind due to inconsistent habits" from "behind but habits are solid, adaptation takes time"
- Never provide medical diagnoses or clinical recommendations

### 12.4 JSON Output Handling

Morning card, check-in, and weekly review calls expect structured JSON. The parsing pattern:

```swift
if let jsonData = raw.data(using: .utf8),
   let parsed = try? JSONDecoder().decode(ResponseType.self, from: jsonData) {
    // use structured data
} else {
    rawFallbackText = raw  // display raw text to user
}
```

Never crash on malformed LLM output. The fallback card shows raw text with a "Retry" button.

### 12.5 Check-In Instruction (Anti-Repetition Rules)

The `checkInInstruction` includes explicit rules to prevent suggesting completed activities:

1. Cross-reference morning card's protocol list against current step count
2. Any `IMPORTANT:` line in the context is a hard constraint the LLM must obey
3. Suggest only what has NOT been done — good options listed explicitly (NSDR, breathwork, hydration, caffeine timing, social connection, focus work)
4. Acknowledge completed activity before making a new suggestion

## 14. Safe Zone Alert System

Evaluated every time the app opens (in `bootstrap()`) and whenever a morning card is generated. Results stored in-memory only — recomputed each session.

### 14.1 Alert Banner

`AlertBannerView` appears at the top of the Today tab scroll view when any metric is in warning or alert state. Each entry shows:
- Metric name
- Current value vs personal baseline
- Percentage or absolute delta
- Status indicator (warning = orange, alert = red)

### 14.2 Evaluation Logic

```swift
// HRV (relative)
let pct = (hrv - avg30) / avg30
if pct < -0.25 → .alert
if pct < -0.15 → .warning

// Resting HR (absolute)
if rhr >= 63 → .alert
if rhr >= 60 → .warning

// Respiratory Rate (absolute)
if resp >= 19 → .alert
if resp >= 17 → .warning
```

## 16. Known Constraints & Edge Cases

**These are explicitly handled — not left as TODOs:**

| Situation | Handling |
|---|---|
| Cold baseline (< 7 days) | LLM instructed to use population reference ranges and temper confidence. Pace assessment not shown in Progress tab. Protocol matcher still returns results via `well_recovered` default. |
| Missing metrics (nil HealthKit data) | Emitted as `"not available (Watch not worn or data not recorded)"`. Never omitted, never zero. |
| No Apple Watch / HRV absent | `well_recovered` flag fires unconditionally when no negative flags are present, ensuring protocols always match. |
| HealthKit permission denied | Handled gracefully per metric. `fetchTodaySummary()` uses `try?` on individual queries; missing values remain nil. |
| LLM JSON parse failure | Raw text shown with fallback card and "Retry" button. Never crashes. |
| API failure (network, rate limit, bad key) | `AppError.apiFailure` shown with user-friendly message and retry button. Raw API error never shown. |
| First app launch | All stores initialize with empty/default values if files don't exist. TrendStore: empty history. GoalStore: empty goals. ProgressStore: empty. |
| Morning card already generated today | `MorningCardStore.loadIfToday()` restores it immediately. "Generate" button not shown again until tomorrow. |
| Daily log already submitted today | Checked via UserDefaults date key in `bootstrap()`. Compact summary shown; form never re-displayed. |
| Check-in suggesting already-done activity | Step count → walk duration estimate. If ≥ 10 min estimated: `IMPORTANT:` hard constraint in context. Instruction explicitly tells LLM to cross-reference morning protocols against completed activity. |
| Tab navigation timing (pending chat) | Both `.onAppear` and `.onChange` handle `PendingChatContext`. `.onAppear` catches first-render case where SwiftUI batches state changes before view hierarchy is set. |
| Chat keyboard blocking tab navigation | `.scrollDismissesKeyboard(.immediately)` + `@FocusState` + keyboard toolbar "Done" button. |
| Stale activity data in chat | `currentSummary` refreshed from HealthKit at the start of each `send()` call. |

## 18. Reference Ranges

Used during cold baseline period and shown in goal setup UI for guidance.

| Metric | Low | Average | Good | Excellent |
|---|---|---|---|---|
| HRV (ms) | < 30 | 40–60 | 60–80 | > 80 |
| Resting HR (bpm) | — | 60–80 | 50–60 | < 50 |
| Sleep duration (hours) | < 6 | 6–7 | 7–8 | 8–9 |
| Sleep efficiency (%) | < 75 | 75–84 | 85–90 | > 90 |
| Respiratory rate (sleep, br/min) | — | 12–20 | 12–16 | 12–14 |

HRV ranges are illustrative. Personal baseline is always the primary reference once ≥ 7 days of data is established.
